title: "Running Tests"

Rue provides a pytest-inspired CLI for running tests, filtering by tags or keywords, controlling concurrency, and reporting results.

This page covers **how to run tests** and **how the reporting system works**, referencing where the behavior lives in the codebase.

## Basic Usage

Run all discovered tests in the current directory:

```bash
uv run rue run
```

After installation, the same entrypoint is also available as `rue run`.
The command name is explicit; bare `rue` shows command help.

Run tests from specific paths:

```bash
uv run rue run tests/
uv run rue run test_chatbot.py test_agent.py
```

## Filtering Tests

### By Keyword Expression

Use `-k` to filter tests by name with boolean expressions:

```bash
# Run tests with "chatbot" in the name
uv run rue run -k chatbot

# Boolean operators: and, or, not
uv run rue run -k "chatbot and not slow"
uv run rue run -k "gpt4 or claude"

# Grouping with parentheses
uv run rue run -k "(fast or smoke) and not flaky"
```

Keyword matching is substring-based: `-k agent` matches `test_agent_response`, `test_weather_agent`, etc.

Rue still applies its own collection rules after path and keyword filtering, so in mixed files it will run only the tests marked with Rue decorators.

### By Tags

Use `-t`/`--tag` to include tests with specific tags:

```bash
# Run tests tagged "smoke"
uv run rue run --tag smoke

# Multiple tags (OR logic)
uv run rue run --tag smoke --tag fast
```

Use `--skip-tag` to exclude tests:

```bash
# Skip slow tests
uv run rue run --skip-tag slow

# Skip multiple tags
uv run rue run --skip-tag slow --skip-tag flaky
```

**Combine filters**:

```bash
# Run fast smoke tests about chatbots
uv run rue run --tag smoke --tag fast -k chatbot
```

## Controlling Execution

### Stop on Failure

**`--maxfail N`** - Stop after N failures:

```bash
uv run rue run --maxfail 3  # Stop after 3 failures
```

**`--fail-fast`** - Stop at the first failed assertion within a test:

```bash
uv run rue run --fail-fast
```

Without `--fail-fast`, Rue collects all assertion failures in a test. With it, the test stops at the first failure.

### Concurrency

Control parallel test execution with `--concurrency`:

```bash
# Sequential (default)
uv run rue run --concurrency 1

# 5 concurrent tests
uv run rue run --concurrency 5

# Unlimited (capped at 10)
uv run rue run --concurrency 0
```

**When to use concurrency**:
- **Sequential (1)**: Default. Predictable output, easier debugging.
- **Concurrent (>1)**: Faster runs for independent tests. Use with stateless SUTs.
- **Unlimited (0)**: Maximum parallelism for large test suites.

Rue plans work in queue stages:
- `ExecutionBackend.MAIN` tests run on the main thread in discovery order and block all other test execution.
- `ExecutionBackend.MODULE_MAIN` tests block only their own module queue.
- `ExecutionBackend.ASYNCIO`, `ExecutionBackend.MODULE_MAIN`, and `ExecutionBackend.SUBPROCESS` tests share the same `--concurrency` budget.

For synchronous tests (`def test_*`), the default `ASYNCIO` backend runs test bodies in worker threads to keep the event loop responsive. Use `@rue.test.backend(ExecutionBackend.MODULE_MAIN)` for a module-scoped queue barrier, and `@rue.test.backend(ExecutionBackend.MAIN)` only when a sync test must run on the main event-loop thread.

### Timeout

Set a global timeout for the entire suite execution:

```bash
# Stop after 300 seconds (5 minutes)
uv run rue run --timeout 300
```

The timeout applies to the entire suite execution, not individual tests.

Timeout is cooperative: when the timeout is reached, Rue marks the suite as stopped early and stops starting new tests, but in-flight work may not stop immediately (especially synchronous tests already running in worker threads).

### Verbosity

Control output detail with `-v` (verbose) or `-q` (quiet):

```bash
# Minimal output (only failures)
uv run rue run -q

# Very minimal
uv run rue run -qq

# Verbose output
uv run rue run -v

# Very verbose
uv run rue run -vv
```

**Verbosity levels**:
- `-qq` or lower: Only failed/errored tests shown
- `-q`: Less output
- Default (0): Standard output
- `-v`, `-vv`: More detail

### Output Capture

Rue captures stdout and stderr only while a wrapped `SUT` call is running.
Assert against `sut.stdout`, `sut.stderr`, or `sut.captured_output` inside your test.
Captured SUT output is kept on the SUT API instead of being printed live by Rue.
Output written outside an active wrapped `SUT` call is not stored on a Rue API.

## OpenTelemetry

Rue enables OpenTelemetry capture by default so spans from your SUT and tests are recorded automatically:

```bash
# Default behavior
uv run rue run

# Disable OpenTelemetry for this suite execution
uv run rue run --no-otel

# Extra processors are for custom SuiteEventsProcessor instances
uv run rue run --processor CustomProcessor
```

In the usual fresh-process setup, `uv run rue run` already writes JSON traces under `.rue/traces/<suite_execution_id>/<test_execution_id>.json` because the CLI attaches `OtelReporter` when OpenTelemetry is enabled. Each file stores `suite_execution_id`, `test_execution_id`, and a nested `trace` tree rooted at the test span.

**Use OpenTelemetry spans for**:
- Asserting tool calls in agent tests
- Debugging LLM request/response flows
- Performance analysis

See [SUT](/concepts/sut) for OpenTelemetry span assertions.

## Custom `suite_execution_id`

By default, Rue generates a `suite_execution_id` automatically. You can provide
one explicitly when you need a stable external correlation ID (for example,
linking CI jobs to Rue suite executions).

### CLI

Provide a UUID with `--suite-execution-id`:

```bash
uv run rue run --suite-execution-id 3f5f5e9a-1c2d-4b5f-9c2b-7f6d8a9b0c1d
```

If that UUID already exists in the configured Turso database, the command exits with code `2`
and no tests are executed.

### Python API

Programmatic suite executions receive their `suite_execution_id` from `SuiteContext`:

```python
from uuid import UUID
from rue.config import Config
from rue.context.runtime import SuiteContext
from rue.events import SuiteEventsReceiver
from rue.cli.rendering.terminal import TerminalSuiteReporter
from rue.resources import DependencyResolver, registry
from rue.storage import TursoSuiteRecorder, TursoSuiteStore
from rue.testing import ExecutableSuite

config = Config()
store = TursoSuiteStore(config.database_path)
store.initialize()

with SuiteContext(
    config=config,
    suite_execution_id=UUID("00000000-0000-0000-0000-000000000001"),
) as context, SuiteEventsReceiver([
    TerminalSuiteReporter(),
    TursoSuiteRecorder(),
]):
    suite = await ExecutableSuite(
        items=items,
        suite_execution_id=context.suite_execution_id,
        resolver=DependencyResolver(registry),
    ).execute()
```

`suite_execution_id` values are currently configured only via CLI
`--suite-execution-id` or `SuiteContext`. They are not read from
`pyproject.toml`, `rue.toml`, or environment variables.

## Configuration Files

Define default options in `pyproject.toml` or `rue.toml`:

**pyproject.toml**:

```toml
[tool.rue]
test-paths = ["tests", "integrations"]
include-tags = ["smoke"]
exclude-tags = ["slow", "flaky"]
maxfail = 5
fail-fast = true
verbosity = 1
concurrency = 4
```

**rue.toml**:

```toml
test_paths = ["tests"]
verbosity = 1
concurrency = 4
```

**Precedence**: CLI args override config files. Config files are discovered by walking up the directory tree from the current working directory.

## Understanding Test Output

Rue reports test status as tests complete with a compact line per file by default:

```
===== RUE SUITE STARTS =====
platform macOS-14.6.1-arm64-arm-64bit -- python 3.12.2 -- rue 0.9.1
rootdir: /Users/you/workspace/project
suite_execution_id: 3f5f5e9a-1c2d-4b5f-9c2b-7f6d8a9b0c1d
git: main (abc12345) dirty

Collected 12 tests

tests/unit/test_chatbot.py ✓✓✗
tests/unit/test_db.py !-
tests/unit/test_known_issues.py x

===== SUMMARY =====
suite_execution_id: 3f5f5e9a-1c2d-4b5f-9c2b-7f6d8a9b0c1d | 6 passed, 1 failed, 1 errors, 1 skipped, 1 xfailed in 892ms
==================
```

Use `-v` to show per-test lines with durations and detailed sub-results.

**Status symbols**:
- `✓` (green): PASSED - Test succeeded
- `✗` (red): FAILED - Assertion failed
- `!` (yellow): ERROR - Unexpected exception
- `-` (yellow): SKIPPED - Test was skipped
- `x` (blue): XFAILED - Expected failure occurred
- `!` (magenta): XPASSED - Expected failure passed (usually bad)

**Exit codes**:
- `0`: All tests passed (or only skipped/xfailed)
- `1`: At least one test failed or errored
- `2`: Invalid CLI usage or configuration error (including duplicate `--suite-execution-id`)

### Repeated Tests

For tests with `@rue.test.iterate()`, verbose output shows aggregated results:

```
  ✓ test_llm_consistency (1234.5ms) PASSED
  ✗ test_flaky_api (567.8ms) FAILED
```

Child test execution results are attached to
`ExecutedTest.sub_test_executions`.

## Suite Event Processor System

Rue uses async suite event processors for output, trace persistence, and other lifecycle integrations.

### The Processor Interface

Processors subclass `SuiteEventsProcessor`:

```python
class SuiteEventsProcessor:
    def configure(self, config: Config) -> None:
        """Adjust processor parameters from runtime config."""

    async def on_suite_execution_start(self, suite: ExecutedSuite) -> None:
        """Called when suite execution starts."""

    async def on_no_tests_found(self, suite: ExecutedSuite) -> None:
        """Called when test collection finds no tests."""

    async def on_collection_complete(
        self,
        items: list[LoadedTestDef],
        suite: ExecutedSuite,
    ) -> None:
        """Called after test collection completes."""

    async def on_test_execution_start(
        self,
        test: ExecutableTest,
        suite: ExecutedSuite,
    ) -> None:
        """Called before test execution starts."""

    async def on_test_execution_complete(
        self,
        execution: ExecutedTest,
        suite: ExecutedSuite,
    ) -> None:
        """Called when any test execution node completes."""

    async def on_suite_execution_complete(self, suite: ExecutedSuite) -> None:
        """Called after suite execution completes."""
```

### Processor Selection

Processor selection follows this public precedence order:
- `TerminalSuiteReporter` is attached by the CLI for normal `rue run` output
- `OtelReporter` is attached when OpenTelemetry is enabled
- CLI `--processor` flags replace the configured custom processor list
- `config.processors` adds custom processors when CLI flags are absent

Official Rue entrypoints append the database recorder so suite execution data streams into Turso as test executions complete. Programmatic suite executions attach one or more processors by opening `SuiteEventsReceiver([...])` around `ExecutableSuite`.

### Built-in Processors

In a normal fresh process, Rue's CLI processor set includes `TerminalSuiteReporter` and `OtelReporter`.

**TerminalSuiteReporter**: Outputs to terminal with Rich formatting.

```python
from rue.cli.rendering.terminal import TerminalSuiteReporter

processor = TerminalSuiteReporter(verbosity=1)
```

**OtelReporter**: Persists OpenTelemetry traces under `.rue/traces`.

```python
from rue.telemetry.otel import OtelReporter

processor = OtelReporter()
```

**TursoSuiteRecorder**: Streams suite execution and test execution data into Turso.

```python
from rue.storage import TursoSuiteRecorder, TursoSuiteStore

store = TursoSuiteStore()
store.initialize()
processor = TursoSuiteRecorder()
```

### Creating Custom Processors

Create custom processors by subclassing `SuiteEventsProcessor`.
Override `on_test_execution_start` if you need live in-flight state. Finished telemetry arrives on `execution.telemetry_artifacts` inside `on_test_execution_complete`:

```python
import json
from pathlib import Path

from rue.config import Config
from rue.events import SuiteEventsProcessor
from rue.telemetry import OtelTraceArtifact
from rue.testing import ExecutedSuite, LoadedTestDef, ExecutedTest

class JsonFileProcessor(SuiteEventsProcessor):
    """Custom processor that writes results to JSON file."""

    def __init__(self, output_path: str):
        self.output_path = Path(output_path)
        self.results = []
        self.verbosity = 0

    def configure(self, config: Config) -> None:
        self.verbosity = config.verbosity

    async def on_no_tests_found(self, suite: ExecutedSuite) -> None:
        _ = suite
        print("No tests found")

    async def on_collection_complete(
        self,
        items: list[LoadedTestDef],
        suite: ExecutedSuite,
    ) -> None:
        _ = suite
        print(f"Collected {len(items)} tests")

    async def on_test_execution_start(
        self,
        test,
        suite: ExecutedSuite,
    ) -> None:
        _ = test, suite
        # Optional lifecycle hook for live/in-flight reporting
        pass

    async def on_test_execution_complete(
        self,
        execution: ExecutedTest,
        suite: ExecutedSuite,
    ) -> None:
        _ = suite
        # Record each test result
        self.results.append({
            "name": execution.definition.spec.full_name,
            "status": execution.result.status.value,
            "duration_ms": execution.result.duration_ms,
            "passed": execution.result.status.value == "passed"
        })
        for artifact in execution.telemetry_artifacts:
            if isinstance(artifact, OtelTraceArtifact):
                print(
                    "Collected OpenTelemetry trace for test execution "
                    f"{execution.test_execution_id}: {artifact.test_execution_id}"
                )

    async def on_suite_execution_complete(self, suite: ExecutedSuite) -> None:
        # Write all results to JSON file
        output = {
            "suite_execution_id": str(suite.suite_execution_id),
            "total": suite.result.total,
            "passed": suite.result.passed,
            "failed": suite.result.failed,
            "duration_ms": suite.result.total_duration_ms,
            "tests": self.results
        }

        self.output_path.write_text(json.dumps(output, indent=2))
        print(f"Results written to {self.output_path}")

    async def on_suite_stopped_early(
        self,
        failure_count: int,
        suite: ExecutedSuite,
    ) -> None:
        _ = suite
        print(f"Suite stopped after {failure_count} failures")
```

### Using Multiple Processors

Use multiple processors simultaneously with the programmatic API:

```python
import asyncio
from rue.config import Config
from rue.context.runtime import SuiteContext
from rue.events import SuiteEventsReceiver
from rue.testing import ExecutableSuite
from rue.cli.rendering.terminal import TerminalSuiteReporter
from rue.telemetry.otel import OtelReporter
from rue.resources import DependencyResolver, registry
from rue.storage import TursoSuiteRecorder, TursoSuiteStore
from my_processors import JsonFileProcessor, SlackProcessor

async def main():
    config = Config()
    store = TursoSuiteStore(config.database_path)
    store.initialize()
    db_writer = TursoSuiteRecorder()
    console = TerminalSuiteReporter(verbosity=1)
    otel = OtelReporter()
    json_processor = JsonFileProcessor("results.json")
    slack = SlackProcessor(webhook_url="https://...")

    with SuiteContext(config=config) as context, SuiteEventsReceiver([
        console,
        otel,
        db_writer,
        json_processor,
        slack,
    ]):
        result = await ExecutableSuite(
            items=items,
            suite_execution_id=context.suite_execution_id,
            resolver=DependencyResolver(registry),
        ).execute()

    print(f"Results: {result.result.passed}/{result.result.total} passed")

asyncio.run(main())
```

## Database Persistence

Rue automatically persists suite execution data to a Turso database for historical tracking and analysis.

### Database Location

By default, Rue stores the database at the project root:

```
<project-root>/.rue/rue.turso.db
```

The database location is determined by walking up from the current directory to find the first directory containing `pyproject.toml`.

### Custom Database Path

Database persistence is required for normal `rue run` suite executions. Configure a custom database location in `pyproject.toml`:

```toml
[tool.rue]
database-path = "/path/to/custom/rue.turso.db"
```

### Database Management Commands

Rue provides CLI commands to inspect, initialize, and reset the database:

#### Check Database Status

View current schema state:

```bash
uv run rue db status
```

**Output:**
```
Database: /Users/you/project/.rue/rue.turso.db
Exists: True
Schema version: <current schema version>
Target version: <target schema version>
Stored suite executions: <count>
Status: Ready
```

#### Initialize Database

Create the database schema:

```bash
uv run rue db init
```

#### Reset Database

Delete and recreate the database (destructive):

```bash
uv run rue db reset --yes
```

Without `--yes`, Rue prints a warning and exits without resetting the database.

### What's Stored

The database stores:
- Suite execution metadata (timestamp, environment, git info)
- Individual test executions and results
- Assertion results and predicate outcomes
- Metric results and statistical data
- Error tracebacks for failed tests

OpenTelemetry trace payloads and trace IDs are stored separately under `.rue/traces` when an `OtelReporter` is attached. Turso storage does not persist them.

### Database Schema

The schema is versioned. Current tables include:
- `suite_executions` - Suite execution records
- `test_executions` - Individual test execution records
- `test_execution_tags` - Tags attached to individual test execution records
- `metrics` - Aggregated metrics
- `assertions` - Assertion outcomes
- `predicates` - Predicate results linked to assertions

Schema version is stored in the `rue_schema` table (shown by `uv run rue db status` as "Schema version").
