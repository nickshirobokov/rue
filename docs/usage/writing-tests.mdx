title: "Writing Tests"

Rue is intentionally “pytest-shaped”: you write plain Python, Rue discovers `test_*.py` modules, collects only explicitly marked Rue tests, injects dependencies by parameter name (like pytest fixtures), runs them, and reports results.

This page focuses on **how to write tests** and (importantly) **where the behavior lives in the codebase**, so you can trust what’s happening.

## TLDR - If you know pytest, you already know 80%

| pytest | rue |
| --- | --- |
| `def test_*(): ...` | `@rue.test` + `def test_*(): ...` |
| `@pytest.fixture` | `@rue.resource` |
| `@pytest.mark.parametrize(...)` | `@rue.test.iterate.params(...)` |
| `@pytest.mark.skip(...)` | `@rue.test.tag.skip(...)` |
| `@pytest.mark.xfail(...)` | `@rue.test.tag.xfail(...)` |
| `@pytest.mark.repeat(3)` | `@rue.test.iterate(3)` |

## Test Discovery

Rue follows pytest-style discovery patterns to find test functions in your codebase:

<Info>
  **Related docs**: [Test (Concept)](/concepts/test), [Testing (API)](/apis/testing)
</Info>

**Files**: Rue discovers Python files matching `test_*.py`:
- `test_chatbot.py` ✓
- `test_agent.py` ✓
- `helpers.py` ✗

Rue also recognizes `conftest.py` and `confrue_*.py` files as shared setup modules:
- `conftest.py` ✓ Loaded as setup
- `confrue_shared.py` ✓ Loaded as setup
- `conftest.py` ✗ Not collected as a test module
- `confrue_shared.py` ✗ Not collected as a test module

For each selected test module, Rue loads ancestor setup files before importing the `test_*.py` file itself. Ancestors load from the suite root down to the test directory. Within each directory, Rue loads `conftest.py` first, then `confrue_*.py` files in lexicographic order.

**Functions**: Inside discovered files, Rue collects only tests marked by Rue decorators. Use `@rue.test` for a plain Rue test, or use another Rue test decorator such as `@rue.test.iterate...` or `@rue.test.tag...`:

```python
import rue

@rue.test
def test_weather_agent():  # ✓ Rue collects this
    pass

def test_pytest_only():      # ✗ Rue ignores this, pytest can still collect it
    pass

def helper_function():       # ✗ Not discovered
    pass
```

**Classes**: Class names no longer matter. Rue collects methods from a class only when the class itself is Rue-decorated, or when individual methods are Rue-decorated:

```python
import rue

@rue.test
class CustomerSupport:
    def test_greeting(self):    # ✓ Collected because the class is marked
        pass

    @rue.test
    def helper(self):            # ✓ Collected because the method is marked
        pass
```

This makes mixed pytest/Rue files possible: pytest can still collect plain `test_*` functions, while Rue only collects the decorated ones.

### Shared Setup With `conftest.py` and `confrue_*`

Use `conftest.py` or `confrue_*.py` for reusable setup that should apply to a directory tree, especially shared helpers and `PROCESS` resources.

```text
project/
├── conftest.py
├── confrue_root.py
├── test_smoke.py
└── api/
    ├── conftest.py
    ├── confrue_api.py
    └── test_orders.py
```

When Rue imports `api/test_orders.py`, it loads:
- `conftest.py`
- `confrue_root.py`
- `api/conftest.py`
- `api/confrue_api.py`
- `api/test_orders.py`

Keep actual tests in `test_*.py` files. Keep shared setup and directory-specific overrides in `conftest.py` or `confrue_*.py` files.

</Info>

## Modifiers

Modifiers are **decorators** that change how a test is *collected* and/or *executed*. Some modifiers expand a single test into many cases (like parametrization or dataset iteration), while others change execution semantics (like repeating a case, or marking it as skipped/xfail). These decorators already mark the test for Rue collection, so you do not need to stack `@rue.test` on top of them.

<Info>
  **Related docs**: [Testing (API)](/apis/testing), [Case (Concept)](/concepts/case)
</Info>

**`@rue.test.iterate.params(names, values)`** - Run the same test with different inputs

```python
@rue.test.iterate.params("model,temp", [("gpt-4", 0.7), ("claude-3", 0.5)])
def test_model_response(model: str, temp: float, chatbot):
    response = chatbot.generate(model=model, temperature=temp)
    assert response
```

**`@rue.test.iterate.cases(*cases, min_passes=len(cases))`** - Iterate over `Case` objects from external sources, optionally allowing pass thresholds

```python
import json
from rue import Case

cases = [Case(**item) for item in json.load(open("cases.json"))]

@rue.test.iterate.cases(*cases)
def test_from_dataset(case: Case, classifier):
    result = classifier(**case.inputs)
    if case.references:
        assert result == case.references["some_ref_value"]

@rue.test.iterate.cases(*cases, min_passes=8)  # Requires at least 8 case passes
def test_from_dataset_threshold(case: Case, classifier):
    result = classifier(**case.inputs)
    assert result == case.references["some_ref_value"]
```

**`@rue.test.iterate.groups(*groups, min_passes=len(groups))`** - Iterate over `CaseGroup` objects with per-group thresholds, group-level references, and an outer group threshold

```python
from rue import Case, CaseGroup

geography = CaseGroup(
    name="geography",
    cases=[
        Case(inputs={"prompt": "Capital of France?"}, references={"expected": "Paris"}),
        Case(inputs={"prompt": "Capital of Germany?"}, references={"expected": "Berlin"}),
    ],
    min_passes=2,
)

music = CaseGroup(
    name="music",
    cases=[Case(inputs={"prompt": "Best rock band?"}, references={"expected": "Metallica"})],
    min_passes=1,
)

@rue.test.iterate.groups(geography, music, min_passes=1)
def test_chatbot(group: CaseGroup, case: Case, chatbot):
    response = chatbot.instance(**case.inputs)
    assert case.references["expected"] in response
```

**`@rue.test.tag(*tags)`** - Organize and filter tests by tags

```python
@rue.test.tag("smoke", "fast")
def test_health_check(api):
    assert api.health_check()

# Run: uv run rue tests run --tag smoke
```

**`@rue.test.tag.skip(reason=...)`** - Skip tests unconditionally

```python
@rue.test.tag.skip(reason="Feature not implemented")
def test_upcoming():
    pass

@rue.test.tag.skip(reason="Requires API key")
def test_external_api():
    pass
```

**`@rue.test.tag.xfail(reason=..., strict=False)`** - Mark tests expected to fail

```python
@rue.test.tag.xfail(reason="Known bug #123")
def test_known_issue():
    assert False  # Won't fail the suite

@rue.test.tag.xfail(reason="Should still fail", strict=True)
def test_strict():
    pass  # If this passes, suite FAILS
```

**`@rue.test.iterate(n, min_passes=n)`** - Run tests multiple times to see if AI behavior is consistent

```python
@rue.test.iterate(10)  # All 10 must pass
def test_consistent(llm):
    assert "hello" in llm.generate("Say hello")

@rue.test.iterate(10, min_passes=8)  # 8 out of 10
def test_mostly_correct(llm):
    assert "hola" in llm.generate("Say hello in Spanish")
```

**`@rue.test.backend(...)`** - Choose how a test executes

By default, Rue treats tests as `ExecutionBackend.ASYNCIO`: `async def test_*` functions are awaited on the event loop, and synchronous `def test_*` functions are offloaded to a worker thread via `asyncio.to_thread(...)`.

Use `ExecutionBackend.MAIN` when a test must run on the main event-loop thread and block all other tests until it finishes. Use `ExecutionBackend.MODULE_MAIN` when a test should block only its own module queue while still running like a normal in-process test. Use `ExecutionBackend.SUBPROCESS` to isolate the test in a worker process.

```python
import rue
import threading
from rue import ExecutionBackend

@rue.test
def test_default_threaded():
    # Runs in a worker thread by default.
    assert threading.current_thread() is not threading.main_thread()

@rue.test.backend(ExecutionBackend.MAIN)
def test_main_thread_only():
    # Runs on the main event-loop thread and acts as a queue barrier.
    assert threading.current_thread() is threading.main_thread()

@rue.test.backend(ExecutionBackend.MODULE_MAIN)
def test_module_barrier():
    # Blocks only this module's queue and still runs in a worker thread.
    assert threading.current_thread() is not threading.main_thread()

@rue.test.backend(ExecutionBackend.SUBPROCESS)
def test_isolated_worker():
    # Runs in a worker process.
    assert True
```

## Resources

Resources are the Rue equivalent of pytest fixtures: named, injectable dependencies that Rue resolves by parameter name.

<Info>
  **Related docs**: [Resource (Concept)](/concepts/resource), [Metric (Concept)](/concepts/metric), [SUT (Concept)](/concepts/sut), [OpenTelemetry Traces (API)](/apis/open-telemetry-traces)
</Info>

**`@rue.resource(scope="test")`** - Define injectable dependencies with lifecycle management

```python
@rue.resource
def database():
    conn = connect_db()
    yield conn  # Injected into tests
    conn.close()  # Automatic cleanup

@rue.resource(scope="process")
def ml_model():
    return load_model()  # Shared across the run for the active provider

@rue.test
def test_query(database, ml_model):
    # Both injected automatically
    result = database.query("SELECT 1")
    prediction = ml_model.predict(result)
    assert prediction
```

Scopes: `"test"` (default), `"module"`, `"process"`.

`PROCESS` resources are hierarchical: same-name providers can coexist, and Rue selects the nearest ancestor provider relative to the requesting test module. Without an active test context, resolution falls back to the flat/latest registered definition.

In practice, put shared `PROCESS` resources in `conftest.py` or `confrue_*.py` files so directory structure controls which override is active.

**`@rue.resource.metric(scope="process")`** - Define a metric as a scoped, injectable measurement object

Metrics behave like resources, but they’re intended to accumulate measurements across many cases and then assert on aggregates at the end of their scope.

The most common pattern is to inject a `Metric` into your tests and use the `metrics(...)` context manager to record assertion outcomes into that metric.

```python
from rue import Metric, metrics

@rue.resource.metric
def accuracy():
    metric = Metric()
    yield metric
    assert metric.mean > 0.8  # Check after all data collected

@rue.test.iterate.params("input,expected", [("a", 1), ("b", 2)])
def test_classifier(input, expected, classifier, accuracy: Metric):
    result = classifier(input)

    # Assertions inside metrics() are recorded as True/False
    with metrics(accuracy):
        assert result == expected
```

Scopes: `"process"` (default), `"module"`, `"test"`.

**`@rue.resource.sut`** - Register a System Under Test (SUT) factory as an injectable, traceable resource

A SUT is the thing you’re actually evaluating (an agent function, a pipeline, a classifier, a client wrapper, etc.). Declaring a factory with `@rue.resource.sut` makes a resolved `SUT` object injectable and traceable, so you can assert not only on the output, but also on how it behaved internally (for example: tool calls).

<Note>
  The decorated function is a factory. It should return or yield `SUT(...)`. Call `sut.instance(...)` for wrapped callables, or configure `methods=[...]` and call `sut.instance.method(...)` for traced instance methods.
</Note>

```python
from rue import SUT
from demo_app import agent

@rue.resource.sut
def weather_agent():
    return SUT(lambda prompt: agent(prompt, tools=["get_weather"]))

def test_agent_uses_tools(weather_agent):
    result = weather_agent.instance("What's the weather?")

    # Access OpenTelemetry spans for assertions
    root_spans = weather_agent.root_spans
    assert root_spans

    # If you want to assert on tool calls, query LLM spans explicitly.
    # Note: attribute keys come from OpenTelemetry LLM instrumentations.
    tool_names = [
        s.attributes.get("llm.request.functions.0.name")
        for s in weather_agent.llm_spans
        if s.attributes
    ]
    assert "get_weather" in tool_names
```

## Custom Assert

Rue transforms Python's `assert` keyword to provide richer testing capabilities for AI systems. When you run test files through Rue's runner, assertions behave differently than standard Python.

### Continue on failure (default behavior)

By default, Rue **continues running remaining assertions** even after one fails. This is different from standard Python, where the first failed assertion stops execution immediately.

```python
def test_multiple_checks(classifier):
    result = classifier("test input")

    assert result.confidence > 0.8  # Fails
    assert result.label != ""        # Still runs
    assert result.valid              # Still runs
```

All three assertions will be evaluated and reported, even if the first one fails. This behavior lets you see all test failures in a single run rather than fixing them one at a time.

To stop on the first failure, use the `--fail-fast` CLI flag:

```bash
uv run rue tests run --fail-fast
```

### Integration with metrics

When assertions are evaluated inside a `metrics()` context manager, Rue automatically records whether each assertion passed or failed to the specified metrics:

```python
from rue import Metric, metrics

@rue.resource.metric
def accuracy():
    metric = Metric()
    yield metric
    assert metric.mean > 0.8

@rue.test.iterate.params("input,expected", [("a", 1), ("b", 2), ("c", 3)])
def test_classifier(input, expected, classifier, accuracy: Metric):
    result = classifier(input)

    # Assertions inside metrics() are recorded as 1 (pass) or 0 (fail)
    with metrics(accuracy):
        assert result == expected
```

### Only works through Rue's runner

**Important:** Rue's assertion transformation only applies when you run files through Rue's test runner:

```bash
uv run rue tests run test_my_tests.py  # ✓ Transformed assertions
python test_my_tests.py           # ✗ Standard Python behavior
uv run test_my_tests.py           # ✗ Standard Python behavior
```

### Assert messages

Assert messages work as expected and are captured in the `AssertionResult`:

```python
def test_validation(response):
    assert response.status == 200, f"Expected 200, got {response.status}"
    assert "error" not in response.body, "Response contains error"
```
