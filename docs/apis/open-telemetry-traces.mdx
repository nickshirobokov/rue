---
title: "Rue OpenTelemetry Traces"
---

Rue OpenTelemetry capture is **Runner-managed**, while persistence is **processor-managed**. Traces exist only while a test function is executing inside an active OpenTelemetry-enabled test execution created by `Runner` or `uv run rue run`.

## OpenTelemetry Defaults

### CLI

```bash
# OpenTelemetry is enabled by default
uv run rue run

# Disable OpenTelemetry explicitly, even if config enables it
uv run rue run --no-otel

# Force-enable OpenTelemetry if config or environment disabled it
uv run rue run --otel

# Extra processors are for custom RunEventsProcessor instances
uv run rue run --processor CustomProcessor
```

### Config

OpenTelemetry settings can be resolved from `rue.toml`, `[tool.rue]` in `pyproject.toml`, environment variables, CLI args, and `addopts`.

```toml
[tool.rue]
otel = false
processors = ["CustomProcessor"]
```

Environment variable:
- `RUE_OTEL`

CLI flags override config and environment.

### Runner

```python
from rue.config import Config
from rue.context.runtime import RunContext
from rue.events import RunEventsReceiver
from rue.cli.rendering.terminal import TerminalRunReporter
from rue.telemetry.otel import OtelReporter
from rue.testing import Runner

with RunContext(config=Config()), RunEventsReceiver([
    TerminalRunReporter(),
    OtelReporter(),
]):
    runner = Runner()
```

Use `Config(otel=False)` to disable capture programmatically.

The CLI always attaches `TerminalRunReporter`; when `otel` is enabled, it also attaches `OtelReporter`. `config.processors` and `--processor` add custom `RunEventsProcessor` instances by registry name.

## Public API

### `SUT` trace access

OpenTelemetry data is exposed only through `SUT` instances while a test is running.

```python
import rue

@rue.resource.sut
def my_agent():
    return rue.SUT(call_agent)

def test_agent_workflow(my_agent):
    result = my_agent.instance("hello")

    assert result
    assert my_agent.root_spans
    assert my_agent.all_spans
    assert my_agent.llm_spans
```

Available read-only span accessors on `SUT`:
- `root_spans`
- `all_spans`
- `llm_spans`

`root_spans`, `all_spans`, and `llm_spans` return `[]` until that `SUT` has been called in the current execution.

## Behavior

- Rue creates one root span per OpenTelemetry-enabled test execution: `test.<full_name>`.
- Aggregator executions (`iterate`, `iterate.params`, `iterate.cases`, `iterate.groups`) do not get their own OpenTelemetry roots; each child execution gets its own trace.
- `@rue.resource.sut` and `@predicate` write spans only when an active OpenTelemetry session exists.
- Full traces are not exposed to test code during execution; test code can inspect only the subtree of each `SUT`.
- OpenAI and Anthropic client instrumentation is initialized by the OpenTelemetry runtime and captured only when calls belong to an active OpenTelemetry test session.

## Storage

- Capture is enabled by default; persistence still happens only when an `OtelReporter` processor is attached
- In the usual fresh-process setup, `uv run rue run` writes `.rue/traces/<run_id>/<execution_id>.json` because the CLI attaches `OtelReporter` when OpenTelemetry is enabled
- `OtelReporter` stores compact JSON payloads under `.rue/traces/<run_id>/<execution_id>.json`
- `OtelReporter` keeps only the 5 most recent run directories under `.rue/traces`
- Custom OTEL persistence or forwarding belongs in custom processors via `RunEventsProcessor.on_execution_complete(execution, run)` and `execution.telemetry_artifacts`
- File format: one compact JSON trace payload per execution with `run_id`, `execution_id`, and a nested `trace` tree rooted at the test span
- `OtelTraceArtifact.spans` stays flat with the canonical OTEL span payload for artifact consumers
- SQLite does not persist OpenTelemetry trace payloads or trace IDs

When OpenTelemetry is active, Rue records these content-bearing SUT and predicate attributes:
- `sut.input.args`
- `sut.input.kwargs`
- `sut.output`
- `predicate.input.actual`
- `predicate.input.reference`
- `predicate.message`
