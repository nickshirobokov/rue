title: "OpenTelemetry Traces"
---

An OpenTelemetry trace is the structured execution record of a single test execution. In Rue, **each concrete test execution gets its own OpenTelemetry trace**: a tree of spans showing what happened (tools, retrieval steps, LLM calls), when it happened, and how long it took.

For `repeat`, `parametrize`, `iter_cases`, and `iter_case_groups`, each child execution gets its own trace.

<Warning>
This page uses “trace” in the OpenTelemetry sense (spans you can query and assert on), not a Python exception traceback/stacktrace.
</Warning>

Using traces enables:
- Debugging and explaining *why* a test failed (what steps ran, in what order, and how long they took)
- Asserting on **execution behavior**, not just outputs (tool calls happened, retrieval ran, etc.)
- Correlating LLM spans with your SUT spans and custom pipeline steps

## How Rue OpenTelemetry traces are structured

When OpenTelemetry is enabled, Rue wraps each concrete test execution in a root span:
- `test.<full_name>`

Inside that test span, you’ll typically see:
- **SUT spans**: created by [`@rue.sut`](/concepts/sut), named `sut.<sut_name>`
- **Predicate spans**: created by [`@predicate`](/apis/predicates), named `predicate.<predicate_name>`
- **Custom step spans**: created by `rue.otel_span("...")`
- **LLM spans**: auto-instrumented spans whose names usually start with `openai.`, `anthropic.`, or `gen_ai.`

## Enable OpenTelemetry

OpenTelemetry capture is disabled by default. Enable it from the CLI:

```bash
uv run rue test --otel
```

In the usual fresh-process setup, `uv run rue test --otel` also writes local JSON traces under `.rue/traces/<run_id>/<execution_id>.json` because the built-in `OtelReporter` is part of the default reporter set. Those files are separate from SQLite storage.

If you override the reporter list, keep `OtelReporter` attached when you still want local trace files:

```bash
uv run rue test --otel --reporter ConsoleReporter --reporter OtelReporter
```

If you override reporters and omit `OtelReporter`, Rue still captures trace data for the run, but it does not persist local `.rue/traces/...` files.

<Note>
The injected `otel_trace` parameter is only available when OpenTelemetry is enabled. Without `--otel`, resolving `otel_trace` raises at runtime.
</Note>

## Basic Usage

Use `otel_trace` to query spans created during the current test execution:

```python
import rue
from rue import SUT

from demo_app.weather import retrieve_docs
from demo_app.weather import weather_agent as prod_weather_agent


@rue.sut
def weather_agent():
    return SUT(prod_weather_agent)


def test_can_inspect_trace(weather_agent, otel_trace):
    with rue.otel_span("retrieve"):
        docs = retrieve_docs("SF weather")

    with rue.otel_span("generate", {"doc_count": len(docs)}):
        out = weather_agent("What's the weather in SF?", docs=docs)

    # All spans created during this test belong to the same OTEL trace
    spans = otel_trace.get_child_spans()
    assert spans

    # Spans created by @rue.sut
    sut_spans = otel_trace.get_sut_spans(name="weather_agent")
    assert sut_spans[0].attributes.get("rue.sut.name") == "weather_agent"

    # If your test evaluates @predicate functions, filter them manually
    predicate_spans = [
        span
        for span in spans
        if (span.attributes or {}).get("rue.predicate")
    ]

    # Attach extra context to the test root span
    otel_trace.set_attribute("response.length", len(out))
```

## Common patterns

### Assert tool-calling contracts (tool dependency + no loops + permissions)

The main point of OpenTelemetry capture is enforcing **workflow contracts** that matter in production (especially for agents): not just “did we return a good string”, but “did we call the right tools, in the right shape, without runaway loops”.

#### 1. If tool A was called, tool B must also be called

Example contract: “if we called `search`, we must also call `cite_sources`”.

```python
def test_tool_dependency(my_agent, otel_trace):
    my_agent("Find the policy and cite sources")

    tools: list[str] = []
    for span in otel_trace.get_llm_calls():
        attrs = span.attributes or {}
        for key, value in attrs.items():
            if key.startswith("llm.request.functions.") and key.endswith(".name") and value:
                tools.append(str(value))

    if "search" in tools:
        assert "cite_sources" in tools
```

#### 2. Assert there are no tool-calling loops (ABAB..., ABCABC...)

This catches common failure modes like calling the same 2–3 tools in a tight cycle.

```python
def test_no_tool_loops(my_agent, otel_trace):
    my_agent("Solve the task with tools, but don't loop.")

    tools: list[str] = []
    for span in otel_trace.get_llm_calls():
        attrs = span.attributes or {}
        for key, value in attrs.items():
            if key.startswith("llm.request.functions.") and key.endswith(".name") and value:
                tools.append(str(value))

    # Example: ["A","B","A","B",...] or ["A","B","C","A","B","C",...]
    for pattern_len in (2, 3):
        for i in range(0, len(tools) - 2 * pattern_len + 1):
            assert tools[i : i + pattern_len] != tools[i + pattern_len : i + 2 * pattern_len]
```

### Inspect LLM calls

If your SUT triggers instrumented LLM clients, you can locate those spans:

```python
def test_llm_calls_are_captured(my_agent, otel_trace):
    my_agent("hello")

    llm_spans = otel_trace.get_llm_calls()
```

## Recommendations

### 1. Prefer trace assertions for execution guarantees

If correctness depends on *how* the system behaves (e.g., “must call retrieval”, “must call tool X”), asserting on spans is more robust than parsing free-form text output.

### 2. Keep spans high-signal

Create a small number of meaningful steps (`retrieve`, `rerank`, `generate`) rather than wrapping every minor helper function in its own span.

### 3. Be deliberate about content capture

OpenTelemetry spans may include request/response content depending on configuration. See the [OpenTelemetry API docs](/apis/open-telemetry-traces) for `RUE_OTEL_CONTENT` and storage details.
