---
title: "Case"
---

`Case` is a container for a single scenario: **inputs** you will pass to your System Under Test (SUT), plus optional **reference values** you’ll assert against.

Using `Case` enables:
- Iterating one test function over many scenarios with `@rue.iter_cases(...)`
- Grouping related cases with `CaseGroup` and iterating with `@rue.iter_case_groups(...)`
- Keeping inputs in `inputs` and calling SUTs with `sut(**case.inputs)` or `sut(**case.inputs.model_dump())` for typed inputs
- Storing reference data in `references` (typed or untyped)
- Tagging and filtering with `tags`, and attaching context with `metadata`

## Basic Usage

```python
import rue
from rue import Case


def classifier(text: str) -> str:
    return "positive" if "love" in text.lower() else "negative"


case1 = Case(
    inputs={"text": "I love this"},
    references={"expected_label": "positive"},
    tags={"smoke"},
)

case2 = Case(
    inputs={"text": "This is bad"},
    references={"expected_label": "negative"},
    tags={"regression"},
)


@rue.iter_cases(*[c for c in (case1, case2) if "smoke" in c.tags])
def test_classifier(case: Case):
    label = classifier(**case.inputs)
    assert label == case.references["expected_label"]
```

## Case API

### Adding input values

`inputs` stores the keyword arguments for your SUT. `sut_input_values` was renamed to `inputs`.

```python
from rue import Case

case = Case(inputs={"prompt": "Hello"})
```

Use `**case.inputs` for dict inputs. If `inputs` is a `BaseModel`, expand `case.inputs.model_dump()` instead:

```python
reply = chatbot(**case.inputs)
```

### Typed inputs

Use `Case[InputsT, RefsT]` when you want runtime validation and IDE support for both inputs and references.

```python
from pydantic import BaseModel
from rue import Case


class AgentReference(BaseModel):
    expected_keywords: list[str]


class AgentInputs(BaseModel):
    prompt: str


case = Case[AgentInputs, AgentReference](
    inputs=AgentInputs(prompt="Say hello"),
    references=AgentReference(expected_keywords=["hello"]),
)

assert case.inputs.prompt == "Say hello"
assert case.inputs.model_dump() == {"prompt": "Say hello"}
```

### Typed references

Use `Case[dict[str, Any], YourModel]` when you want typed references but plain dict inputs.

```python
from typing import Any

from pydantic import BaseModel
from rue import Case


class AgentReference(BaseModel):
    expected_keywords: list[str]
    min_response_length: int = 10


case = Case[dict[str, Any], AgentReference](
    inputs={"prompt": "Say hello"},
    references=AgentReference(
        expected_keywords=["hello"],
        min_response_length=5
        ),
)

assert case.references.min_response_length == 5
```

### Untyped references

The default `Case` uses untyped references (a dict) and `references` defaults to `{}`:

```python
from rue import Case

case = Case(
    inputs={"x": 2},
    references={"expected": 4},
)

assert case.references["expected"] == 4
```

### Providing ID for persistence

Each case has an `id: UUID` (auto-generated by default). Provide it explicitly when you want stable case IDs across runs or when you store cases in datasets.

Rue stores that value on each child execution as `case_id`, which makes it useful for dataset correlation and history across runs. Human-readable child suffixes come from `repr(case.metadata)` when metadata is present.

```python
from uuid import UUID

from rue import Case


case = Case(
    id=UUID("00000000-0000-0000-0000-000000000001"),
    inputs={"x": 1},
)
```

### Adding metadata

Use `metadata` for extra context and reporting. Values must be JSON-like primitives (`str | int | float | bool | None`).

```python
from rue import Case

case = Case(
    inputs={"prompt": "Hello"},
    metadata={"priority": "high", "latency_budget_ms": 200},
)
```

### Adding tags for filtering

Use `tags` to label cases and then select subsets with normal Python:

```python
from rue import Case

cases = [
    Case(inputs={"x": 1}, tags={"smoke"}),
    Case(inputs={"x": 2}, tags={"regression"}),
]

smoke_cases = [c for c in cases if "smoke" in c.tags]
```

### Validating cases for a SUT

If cases come from files/APIs, validate them against your SUT's signature by attaching them to `@rue.sut(validate_cases=...)`:

```python
import rue
from rue import Case


cases = [
    Case(inputs={"prompt": "Hello", "temperature": 0.5}),
    Case(inputs={"prompt": "Hi"}),
]


@rue.sut(validate_cases=cases)
def my_agent():
    def run(prompt: str, temperature: float = 0.7) -> str:
        return f"{prompt} @ {temperature}"

    return run


@rue.iter_cases(*cases)
def test_my_agent(case: Case, my_agent):
    my_agent(**case.inputs)
```

## CaseGroup

A `CaseGroup` bundles related `Case` objects together with **group-level references** and a **pass threshold** (`min_passes`). This is useful when your evaluation naturally splits into logical groups (e.g. topics, languages, difficulty tiers) and you want to:

- Assert on group-level data that applies to every case in the group
- Set a per-group pass threshold instead of a single global one
- Get hierarchical reporting: run → groups → cases

### Basic Usage

```python
import rue
from rue import Case, CaseGroup


geography = CaseGroup(
    name="geography",
    cases=[
        Case(inputs={"prompt": "Capital of France?"}, references={"expected": "Paris"}),
        Case(inputs={"prompt": "Capital of Germany?"}, references={"expected": "Berlin"}),
    ],
    min_passes=2,  # strict: both must pass
)

music = CaseGroup(
    name="music",
    cases=[
        Case(inputs={"prompt": "Best rock band?"}, references={"expected": "Metallica"}),
        Case(inputs={"prompt": "Best pop artist?"}, references={"expected": "Lady Gaga"}),
    ],
    min_passes=1,  # tolerant: at least one must pass
)


@rue.iter_case_groups(geography, music)
def test_chatbot(group: CaseGroup, case: Case, chatbot):
    response = chatbot(**case.inputs)
    assert case.references["expected"] in response
```

`@rue.iter_case_groups` injects two parameters by name:
- **`group`** — the current `CaseGroup` (with its `name`, `references`, and `cases`)
- **`case`** — the current `Case` inside that group

The test passes only if **every group** meets its own `min_passes` threshold.

### Typed group references

`CaseGroup` accepts three type parameters: case-level inputs, case-level references, and group-level references.

```python
from pydantic import BaseModel
from rue import Case, CaseGroup


class CaseRefs(BaseModel):
    expected: str


class GroupRefs(BaseModel):
    stop_keywords: list[str]


class CaseInputs(BaseModel):
    prompt: str


group = CaseGroup[CaseInputs, CaseRefs, GroupRefs](
    name="geography",
    references=GroupRefs(stop_keywords=["Lol", "Kek"]),
    cases=[
        Case[CaseInputs, CaseRefs](
            inputs=CaseInputs(prompt="Capital of France?"),
            references=CaseRefs(expected="Paris"),
        ),
    ],
)

# IDE autocomplete works on both levels
group.references.stop_keywords   # list[str]
group.cases[0].references.expected  # str
```

### Validation

`CaseGroup` enforces these constraints at creation time:

- `cases` must contain **at least 1** case
- `min_passes` must be **≥ 1**
- `min_passes` cannot exceed the number of `cases`

```python
# All of these raise ValueError at creation time:
CaseGroup(name="empty", cases=[], min_passes=1)              # no cases
CaseGroup(name="zero", cases=[Case(...)], min_passes=0)      # min_passes < 1
CaseGroup(name="over", cases=[Case(...)], min_passes=2)      # min_passes > len(cases)
```

### Validating grouped cases for a SUT

To validate all case inputs against your SUT signature, flatten the groups:

```python
import rue
from rue import CaseGroup

all_groups = [geography, music]

@rue.sut(validate_cases=[case for group in all_groups for case in group.cases])
def chatbot():
    return my_chatbot_fn
```

## Recommendations

### 1. Use Cases when data comes from external sources

If you're hardcoding inputs directly in your test functions, you probably don't need `Case`.

**Don't do this:**
```python
from rue import Case, iter_cases

# Hardcoding simple data in Case objects is unnecessary
cases = [Case(inputs={"x": 1}), Case(inputs={"x": 2})]

@iter_cases(*cases)
def test_simple(case, add_one):
    result = add_one(**case.inputs)
    assert result
```

**Do this:**
```python
import rue

# For simple hardcoded data, use parametrize directly
@rue.parametrize("x", [1, 2, 3, 4, 5])
def test_simple(x, add_one):
    result = add_one(x)
    assert result
```

When loading cases from JSON/APIs, prefer typed cases such as `Case[YourInputs, YourRefs]` when structure matters, and validate inputs up-front with `@rue.sut(validate_cases=...)`.

### 2. Use tags and metadata for dynamic case selection

Structure your cases with tags and metadata to enable flexible selection without changing test code.

```python
import rue
from rue import Case
import os

cases = [
    Case(
        inputs={"text": "short"},
        tags={"smoke", "fast"},
        metadata={"execution_time_ms": 50}
    ),
    Case(
        inputs={"text": "very long input" * 100},
        tags={"regression", "slow"},
        metadata={"execution_time_ms": 5000}
    ),
]

# Run different subsets based on context
if os.getenv("CI_QUICK"):
    # Fast tests only in CI
    test_cases = [c for c in cases if "fast" in c.tags]
else:
    # Full suite locally
    test_cases = cases

@rue.iter_cases(*test_cases)
def test_text_processor(case: Case, processor):
    processor(**case.inputs)
```
